## 二、薄弱点与漏测场景（按优先级）

### P0（必须补）

1. **src/risk/risk_manager.py（0%）**  
   - 漏测：`check_signal`（HOLD/仓位超限/日内亏损）、`update_daily_loss`、`reset_daily`。  
   - **已做**：新增 `tests/test_risk_manager.py`，覆盖上述行为。

2. **OrderExecutor / place_order 调用方式**  
   - 漏测：测试只断言「位置参数第 5 个为 time_in_force」，实际部分路径用 kwargs。  
   - **已做**：`test_order_executor_uses_time_in_force_day` 改为同时接受 args[4] 或 kwargs['time_in_force']`。

3. **BRACKETS 组合单与止损回退（回归）**  
   - **已做**：`tests/test_api_adapter_bracket_and_fill.py` — `_order_status_is_filled`；`place_limit_with_bracket` 生成 LOSS+PROFIT 两腿；组合单失败 → 3 次 `place_order`；`wait_until_buy_filled` 为 False → 仅 1 次主单、不挂 STP/TP。

4. **设计完备性 + DFX 门禁（与文档 §3.4/3.7 对齐，已落地 CI）**  
   - **文档存在性**：`tests/test_design_completeness_dfx_gate.py` — 强制存在 `docs/设计完备性_业界方案与DFX.md`、`docs/需求分析和Feature测试设计.md`、`docs/reports/日志文件在哪儿.md`、`docs/reports/止损未触发_取证与归因.md`；并断言 `order_log.DFX_EXECUTION_FILE` / `log_dfx` 仍在实现中。  
   - **DFX 事件链**：`tests/test_executor_dfx_emission.py` — `OrderExecutor.execute_buy` 组合单成功须写 `bracket_submitted`；组合单失败回退须写 `bracket_failed` + `stop_loss_submitted` + `take_profit_submitted`。  
   - **测试时 DFX 不落生产盘**：`tests/conftest.py` 会话级将 `order_log.DFX_EXECUTION_FILE` 指向临时目录（与 `ORDER_LOG_FILE` 同目录）。  
   - **一键脚本 + CI**：`scripts/verify_design_completeness_and_dfx.py`；`.github/workflows/ci.yml` 步骤「Design completeness + DFX gate」。  
   - **推前/例行命令**：`PYTEST_DISABLE_PLUGIN_AUTOLOAD=1 python scripts/verify_design_completeness_and_dfx.py`

### P1（高优先级）

5. **api_adapter 错误路径与边界**  
   - 漏测：place_order 失败、get_orders 异常、不同 order_type 分支（STP/STP_LMT/MKT/LMT）。  
   - **已做（部分）**：`tests/test_api_adapter_trade.py` — Mock：无 account、Order 对象缺 account、get_orders/get_order；Real：`get_orders`/`get_order` 空参、无 account 时下单失败、LMT/MKT/STP（缺 stop_price）分支（mock `future_contract` + order_utils）。  
   - **待补**：`get_orders`/`get_order` 异常吞掉返回 `[]`/`None`；STP_LMT 参数校验；真实 SDK 异常映射。

6. **tiger1 主循环与 place_tiger_order**  
   - 漏测：747 行未覆盖，含主循环、风控分支、place_tiger_order 实盘/模拟分支、订单日志失败路径。  
   - 建议：对 `place_tiger_order` 做参数化测试（LMT/MKT、有/无止损止盈、mock api 成功/失败）；主循环可做「跑 N 轮无异常」的集成用例。

7. **order_log**  
   - 漏测：log_order、log_api_failure_for_support 的多种参数组合与写文件/缓冲行为。  
   - **已做（部分）**：`tests/test_order_log_behavior.py` — real+success+mock order_id 降级为 fail、纯数字长 ID 保留 success、非法 order_type 归一化、api_failure 与 take_profit/stop_loss 封装（依赖 `conftest` 临时目录）。

8. **OrderExecutor 单测与「本机有 openapicfg」**（防回归）  
   - 只要 `trade_api.account` 或 `t1.client_config.account` 非空，买入/卖出成功路径会做 **get_order/get_orders 后台校验**；若只 Mock `place_order` 而不写入适配器 `orders`，会误报「后台未查到」。  
   - **做法**：该类用例在 setUp 中 **`t1.client_config = None`**（tearDown 恢复），或显式 Mock `get_order`/`get_orders`/`wait_until_buy_filled`；断言「只调 place_order」时需 **`place_limit_with_bracket` 失败或 spec 不含该方法**（组合单优先）。  
   - **可移植路径**：`pytest.ini` 的 `pythonpath = .` + `tests/conftest.py` 将仓库根插入 `sys.path`，新用例避免写死 `/home/cx/tigertrade`。
