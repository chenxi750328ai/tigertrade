# 「昨天没止损」如何取证与归因

**项目责任（DFX）**：自本迭代起，组合单/止损/止盈关键路径会**强制追加** `run/dfx_execution.jsonl`（见 `order_log.log_dfx` 与 `tiger1._dfx_log`、`OrderExecutor` 调用）。**不应**再出现「只能靠用户终端是否重定向」才能归因的情况。若该文件仍为空，说明当日进程**未走到**对应代码路径或未从仓库根启动导致 `run/` 不可写——属部署/权限问题，须单独排查。

**具体原因**仍须用**证据**收敛到一条链：下面列可验证项；**不得**无证据断言。

---

## 1. 先区分两种完全不同的现象

| 现象 | 含义 | 怎么证 |
|------|------|--------|
| **保护单没挂上** | 后台没有对应 STP / 没有 BRACKETS 子单 / 子单被拒 | 订单列表 + 程序日志里的 `止损单提交失败` / `组合单失败` / `跳过交易所止损` |
| **保护单挂着但未成交** | 止损价未被行情触发 | 看止损触发价 vs 当日高低价；订单状态仍为 WORKING |

---

## 2. 从本仓库程序可核对的证据（按优先级）

1. **`run/dfx_execution.jsonl`**（**首选**，DFX 专用）  
   - `bracket_submitted` / `bracket_failed`：组合单是否提交、失败原文  
   - `sl_tp_skipped` + `buy_not_filled_within_timeout`：程序**主动未挂**止损止盈的原因  
   - `stop_loss_submit_failed` / `take_profit_submit_failed`：`detail` 为 **Tiger/API 异常原文**  
   - `stop_loss_submitted` / `take_profit_submitted`：子单已提交及 `child_order_id`

2. **`run/order_log.jsonl`**  
   - 查 `order_type`（含 `limit_bracket`）、`status`、`error`、主单 `order_id`。

3. **运行日志**（`run/*.log` 或终端重定向）  
   检索关键字（**命中哪条，原因就落在哪类**）：  
   - `组合单失败` → 组合单 API 未接受，走了回退  
   - `止损单提交失败` → **后面必须跟 Tiger 返回的异常原文**，才是根因  
   - `超时未 FILLED` / `跳过交易所止损` → 限价主单未在超时内成交，程序**故意**未挂 STP  
   - `已提交止损单` / `已提交 BRACKETS` → 说明至少提交层成功，若仍无保护需查后台是否撤单  

4. **Tiger 客户端 / OpenAPI `get_orders`**  
   - 主买单 `FILLED` 的时间点之后，是否存在 **SELL + STP** 或 **括号单**；状态与 `reason`。

---

## 3. 协作边界

- 远程助手若**拿不到**你本机 `run/dfx_execution.jsonl` / `order_log.jsonl` 的**实际内容**，不能替你在对话里**拍板**唯一根因；应要求你贴出相关行或文件片段。

---

## 4. 建议你一次性贴出的材料（便于一次性定性）

- 出问题当日 **主买订单号**（Tiger 数字 id）  
- **grep 结果**：上述关键字在日志里的**完整一行**（含异常栈或 API 原文）  
- 后台截图或导出：**该主单关联的挂单列表**（含止损/括号单状态）

有了以上任意组合，就可以把原因收敛到**一条可复述的证据链**。
