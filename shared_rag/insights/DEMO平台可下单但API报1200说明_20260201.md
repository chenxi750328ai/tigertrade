# DEMO 平台可下单但 API 报 1200 说明

**时间**: 2026-02-01  
**类别**: insights  
**标签**: demo, api, 1200, futures

---

## 现象

- **DEMO 平台**（网页/APP）：可以下单。
- **API**：`place_tiger_order` 返回 `code=1200 msg=Orders cannot be placed at this moment`。

## 可能原因

1. **品种与交易时段**  
   我们当前下的是 **COMEX 白银期货（SIL2603）**。COMEX 有固定交易时段（美东时间），非交易时段 API 可能统一拒绝期货单；平台端有时会允许挂单（下一时段成交），所以看起来“能下单”。

2. **API 与平台规则不一致**  
   部分券商对「API 下单」和「客户端下单」的规则不同（例如 API 禁止非交易时段下单，客户端允许预挂单）。

3. **账户/合约权限**  
   DEMO 账户在平台上下单的若是股票或其他品种，而不是 SIL2603，则不会触发期货在非时段的限制。

## 代码是否与昨日成功时一致

**是的，未改动。** 失败时的 API 调用路径与昨日成功时一致：`place_tiger_order` → `api_manager.trade_api.place_order(order)`，参数为同一套（合约 SIL2603、方向 BUY、订单类型 LMT、数量 1、限价 91.63、有效期限 DAY 等）。近期仅增加了「订单 LOG 的 source/manual、失败时写 api_failure_for_support」等日志侧改动，**未修改**实际发给老虎的 `place_order` 参数或调用方式。因此 1200 属于**服务端/时段/账户限制**，而非本地代码变更导致。

## 建议

- 在 **COMEX 白银交易时段**（美东工作日对应时段）再用 API 试同一合约（SIL2603）。
- 确认平台上下单的品种与 API 一致（都是 SIL2603）。
- 若仍报 1200，可把 **code=1200、合约、账户、请求时间** 及 `scripts/print_api_failure_for_support.py` 输出的完整订单参数提供给 Tiger 客服确认 API 与期货时段策略。
