## 一、项目角色定义

| 角色 | 职责摘要 | 产出/对接 |
|------|----------|-----------|
| **Leader（PL / 陈正霞）** | 项目整体负责、派活与验收、CI 门禁、问题根因与协调；**监控项目计划执行、各成员进展与问题**。 | 计划与 STATUS 更新、例行检查、协调结论 |
| **设计** | 策略与架构设计、设计文档编写与维护、参数与风控方案、与开发/测试对齐实现与用例。 | 设计文档（如 `docs/strategy_designs/`）、方案说明、参数建议 |

| **测试** | 用例与回归、覆盖率与质量门禁、DEMO/稳定性验证、问题复现与回归防护。 | 测试通过/报告、覆盖率、问题单与回归结论 |

## 二、各角色每日例行工作

### 2.1 Leader（陈正霞 / PL）

**除执行本人例行工作外，必须做：**

1. **监控项目计划执行**
   - 查看 [项目计划_月度周计划](../../docs/项目计划_月度周计划.md) 与 [STATUS](../../docs/STATUS.md)。
   - 核对任务项「计划结束时间」与「实际完成时间」「进展」是否更新；滞后项标注原因与下一步。
   - 必要时更新计划文档的「进展」「实际完成时间」「风险和求助」。

2. **监控成员进展与问题**
   - 收集设计/开发/测试的当日或当周进展（可从日报、RAG、江湖 complete_task / send_message 获取）。
   - 汇总阻塞点、风险与求助；需协调的及时派活或升级。
   - 在 STATUS 或周报中体现「成员进展摘要」与「待解决问题」。

3. **本人例行工作**（与全员清单一致，见下方「全员例行」）
   - 至少覆盖：CI/CD 测试、覆盖率统计、问题解决、20h DEMO 状态查看、收益与算法优化（或委托）、**状态页刷新与推送**。
   - 状态页由 Leader 负责内容与对外一致性；可委托他人更新后回报。

### 2.2 设计

**每日例行：**

1. **设计文档与方案**
   - 按当前迭代维护/更新策略设计文档（如 `docs/strategy_designs/`）、参数说明、风控与流程说明。
   - 新增或大改方案时，在 RAG 或 docs 留记录，并标注「设计文档」链接供开发/测试查阅。

2. **与开发/测试对齐**
   - 确认开发实现与设计一致（接口、参数、行为）；不一致处记录并推动修正或更新设计。
   - 确认测试用例覆盖设计中的关键场景与边界（如风控上限、止损止盈、DEMO 硬顶）。

3. **参与例行中的「设计相关」项**
   - 需要时参与：算法优化中的参数与策略建议、策略报告中的算法说明与设计文档链接维护。
   - 查阅 RAG 中与策略/风控/参数相关的 insights 与 best_practices，避免重复踩坑。

### 2.3 开发

**每日例行：**

1. **CI 与代码质量**
   - 运行与 CI 一致的测试（如 `pytest tests/ -m "not real_api"`），保证通过后再推送。
   - 按需跑覆盖率（如 `pytest --cov=src`），关注本次改动模块的覆盖。

2. **问题解决**
   - 对当日/近期分配的 Bug、失败用例或告警进行修复与验证；根因与解法写入 RAG 或 docs。

3. **DEMO / 数据 / 脚本**
   - 按任务需要：查看或启动 20h DEMO 状态、跑数据刷新或预处理（如 `merge_recent_data_and_train.py`）、跑算法优化脚本（如 `optimize_algorithm_and_profitability.py`），并回报结果。

4. **文档与交付**
   - 代码与行为变更同步更新相关文档（如订单执行流程、DEMO 运维说明）；交付物与计划任务项对齐。

### 2.4 测试

**每日例行：**

1. **测试与覆盖率**
   - 运行完整或约定范围的测试（与 CI 一致），确保无新增失败；执行带覆盖率的测试并查看报告。
   - 对关键路径（订单、风控、DEMO 路径）查漏补缺，保证「真实交易可靠性」有可执行断言（见 RAG：回溯_为何测试与DEMO未发现无止损止盈与超仓）。

2. **稳定性与 DEMO**
   - 参与或监督 20h 稳定性测试 / DEMO 运行；记录运行状态、日志摘要与异常；跑完或定时做异常订单/持仓检查。

3. **问题与回归**
   - 对新发现问题做复现与记录；修复后做回归验证；重要根因与用例补充写入 RAG 或 docs。

4. **与开发/设计对齐**
   - 测试用例与设计场景、需求一致；发现设计未覆盖的边界及时反馈设计/开发。

## 三、全员例行工作（正本）

执行「例行工作」时，**必须覆盖**以下全部项。完整正本与防漏说明见：`shared_rag/best_practices/例行工作清单_agent必读.md`。

1. **CI/CD 测试** — 运行 CI 或等价测试，确保通过。  
2. **覆盖率统计** — 带覆盖率运行测试，关注关键模块。  
3. **问题解决** — 修复本次/近期 Bug 与失败用例，记录根因与解法。  
4. **20 小时稳定性测试（含 20 小时 DEMO 运行）** — 按需运行或查看状态，记录日志与异常。  
5. **数据刷新和模型训练** — 数据刷新与多模型对比训练（如 `train_multiple_models_comparison.py`）。  
6. **收益率分析和算法优化** — 两项都做（如 `optimize_algorithm_and_profitability.py`），产出报告。  
7. **状态页刷新** — 更新 `docs/status.html`，提交并推送；由 Leader 负责，可委托后回报。

**按角色侧重**：设计侧重 5/6 中的参数与方案；开发侧重 1/2/3/4/5/6 的执行与交付；测试侧重 1/2/3/4 与回归；Leader 在完成本人例行基础上，**必须**做计划监控与成员进展/问题汇总。

## 四、私有 RAG 与引用

- **项目私有 RAG**：即本仓库内 `shared_rag/`。所有角色开工前应查阅 `shared_rag/best_practices/` 下与本角色相关的文档（如例行工作清单、状态页刷新规范、设计文档索引）。
- **角色与例行**：本文档为「项目角色定义」与「各角色每日例行工作」的正本；计划、STATUS、日报引用角色或例行时，应指向本文档或 `例行工作清单_agent必读.md`，不得自拟简写版或合并为少于 7 项的全员例行。
